# Failure conditions

The schema should be considered weak or non-viable if one or more of these happens repeatedly:

1. Many cards cannot be classified without arbitrary judgement.
2. Skill levels regularly require another family as the main learning objective.
3. Most useful activities become integrations, leaving pure power cards artificial or trivial.
4. The same capability appears in multiple families with no clear primary learning problem.
5. Cross-cutting families such as Debugging, Design, Safety or Power cannot fit the level ladder.
6. Dependencies form circular chains.
7. The number of integration cards explodes faster than the model can practically manage.
8. The distinction between skill, integration and invention becomes unclear to a facilitator.
9. The schema works only because of subjective exceptions.
10. The schema becomes too complex for a weekly children’s club.

A negative result should include:
- the smallest example that demonstrates the problem
- why the existing schema cannot handle it cleanly
- whether a minor amendment would fix it
- whether the amendment damages the original purpose
